# Team Falcons Unity 项目 Git 分支协作规范

## 1. 协作目标

为了避免多人同时修改 Unity 项目时出现脚本覆盖、场景冲突、Prefab 丢失、`.meta` 文件错乱等问题，团队统一采用：

* `main`：稳定主分支，只保存已经验证可运行的内容。
* `feature/xxx`：功能开发分支，每个人开发新功能时创建。
* Pull Request：功能完成后申请合并到 `main`。
* 禁止长期直接在 `main` 分支开发。

---

## 2. 首次加入项目

### 2.1 获取仓库权限

如果你需要向仓库提交代码或资源，请联系仓库管理员，将你的 GitHub 账号添加为 Collaborator，并授予 Write 权限。

只需要下载和运行项目，则不需要 Write 权限。

---

### 2.2 克隆项目

在你想存放项目的位置打开终端，执行：

```bash
git clone https://github.com/luofeiyu-61/Team-Falcons.git
cd Team-Falcons
```

然后使用 Unity Hub 打开 `Team-Falcons` 文件夹。

首次打开时，Unity 会自动生成 `Library` 文件夹。这是本地缓存，不需要提交到 Git。

---

## 3. 每次开始开发前的标准流程

每次开始新功能前，先执行：

```bash
git switch main
git pull origin main
git switch -c feature/功能名
```

例如，你要开发玩家移动功能：

```bash
git switch main
git pull origin main
git switch -c feature/player-movement
```

你要开发背包系统：

```bash
git switch main
git pull origin main
git switch -c feature/inventory
```

这三步的含义是：

1. 切换回团队主线 `main`。
2. 拉取队友最新合并的内容。
3. 基于最新主线创建自己的功能分支。

不要直接在 `main` 分支上长期修改项目。

---

## 4. 开发过程中的提交方式

开发完成一个小阶段后，及时提交，不要等到整个功能全部完成才提交。

先查看当前修改：

```bash
git status
```

确认无误后：

```bash
git add .
git commit -m "feat: add player movement"
```

常用提交信息格式：

```text
feat: add player movement
feat: add inventory UI
fix: fix player jump bug
fix: fix camera follow issue
refactor: reorganize player controller
art: add environment materials
ui: add main menu interface
docs: update project documentation
```

提交后推送自己的分支：

```bash
git push -u origin feature/player-movement
```

第一次推送分支时需要加 `-u`。之后继续推送只需要：

```bash
git push
```

---

## 5. 功能完成后的合并流程

当你的功能已经完成，并且在 Unity 中运行正常后：

1. 将自己的功能分支推送到 GitHub。
2. 打开 GitHub 仓库页面。
3. 点击 `Compare & pull request`。
4. 创建 Pull Request。
5. 确认合并方向为：

```text
feature/你的功能分支 → main
```

6. 在 Pull Request 描述中说明：

   * 做了什么功能。
   * 修改了哪些主要文件或场景。
   * 是否需要其他成员注意。
   * 是否已经在 Unity 中测试通过。

示例：

```text
本次内容：
- 完成玩家基础移动和跳跃
- 添加第三人称相机跟随
- 修改 PlayerController.cs
- 修改 TestScene 场景

已在 Unity 中测试，角色可正常移动、跳跃和旋转。
```

合并到 `main` 前，最好由另一位成员简单检查一次。

---

## 6. 开发期间如何同步队友的新代码

当其他人的功能已经合并到 `main`，而你还在自己的功能分支开发时，需要把最新主线同步进来。

先确认你当前在自己的功能分支，例如：

```bash
git branch
```

带有 `*` 的分支就是你当前所在分支。

然后执行：

```bash
git fetch origin
git merge origin/main
```

例如你当前在：

```text
feature/player-movement
```

执行后会把最新的 `main` 合并到你的功能分支中。

如果没有冲突，继续开发即可。

如果出现冲突，不要直接乱删文件。先执行：

```bash
git status
```

查看冲突文件，并联系相关修改者确认应该保留哪一部分内容。

---

## 7. Unity 项目必须提交和禁止提交的内容

### 必须提交

以下内容必须进入 Git：

```text
Assets/
Packages/
ProjectSettings/
所有 .meta 文件
.gitignore
README.md
```

特别注意：

```text
.meta 文件必须提交
```

Unity 使用 `.meta` 文件记录资源 ID。缺失 `.meta` 文件可能导致材质、Prefab、脚本引用丢失。

---

### 禁止提交

以下内容是本地缓存、临时文件或构建文件，不应该提交：

```text
Library/
Temp/
Logs/
Obj/
Build/
Builds/
UserSettings/
.vs/
```

仓库中的 `.gitignore` 已经配置了大部分忽略规则。

如果你发现 `Library/`、`Temp/` 等文件被 Git 跟踪，请立即联系管理员处理，不要直接提交。

---

## 8. Unity 场景和 Prefab 的协作规则

Unity 的 `.unity` 场景文件和 `.prefab` 文件在多人同时修改时非常容易冲突。

因此团队遵守以下规则：

### 8.1 同一时间尽量不要多人修改同一个场景

例如：

```text
A 同学：修改 MainScene 的地图布局
B 同学：修改 MainScene 的敌人生成逻辑
```

这种情况容易产生场景文件冲突。

更好的方式是：

```text
A 同学：负责 EnvironmentScene
B 同学：负责 BattleScene
C 同学：负责 UI 场景或独立 Canvas Prefab
```

---

### 8.2 修改公共 Prefab 前先沟通

公共 Prefab 包括：

```text
Player
Enemy
UI Canvas
GameManager
Camera
```

如果需要修改这些核心对象，先在群里说明，避免两个人同时编辑同一个 Prefab。

---

### 8.3 脚本尽量按模块拆分

建议按功能建立目录：

```text
Assets/
├── Scripts/
│   ├── Player/
│   ├── Enemy/
│   ├── UI/
│   ├── System/
│   └── Tools/
├── Prefabs/
├── Scenes/
├── Materials/
├── Animations/
└── Audio/
```

每个人尽量负责独立模块，减少同时修改同一个脚本文件。

---

## 9. 每天结束开发前的检查清单

每天结束开发前，请确认：

```text
[ ] Unity 项目可以正常打开
[ ] 当前场景可以正常运行
[ ] 没有提交 Library、Temp 等缓存文件
[ ] 所有新增资源都包含 .meta 文件
[ ] 已执行 git status 检查修改内容
[ ] 已提交本次修改
[ ] 已 push 到自己的功能分支
```

建议执行：

```bash
git status
git add .
git commit -m "feat: describe your changes"
git push
```

## 10. 常见问题

### Q1：我忘记切分支，直接在 main 上改了怎么办？

先不要 push。

执行：

```bash
git switch -c feature/功能名
```

这样会把你当前的修改带到新分支中，然后再正常提交：

```bash
git add .
git commit -m "feat: your feature"
git push -u origin feature/功能名
```

---

### Q2：我执行 git pull 后出现冲突怎么办？

先执行：

```bash
git status
```

不要直接删除冲突文件。

找到冲突文件后，联系修改该文件的队友，共同确认保留内容。处理完成后：

```bash
git add .
git commit -m "fix: resolve merge conflict"
```

---

### Q3：为什么 Unity 打开项目后生成了很多文件？

Unity 会自动生成：

```text
Library/
Temp/
Logs/
Obj/
```

这些是本地缓存，不需要提交。其他成员打开项目时也会自己生成。

---

### Q4：为什么我拉取代码后 Unity 出现资源丢失或引用丢失？

优先检查：

```text
.meta 文件是否被提交
```

如果 `.meta` 文件丢失，Unity 无法正确识别资源 ID，就可能导致 Prefab、材质、脚本引用断开。

---

## 11. 最终协作原则

```text
1. 开始新功能前，先拉取最新 main。
2. 每个功能都在独立 feature 分支开发。
3. 不要长期直接在 main 上开发。
4. 小步提交，及时 push。
5. 不要提交 Library、Temp、Build 等缓存文件。
6. 所有 .meta 文件必须提交。
7. 修改公共场景、Prefab、核心脚本前先沟通。
8. 功能完成后通过 Pull Request 合并到 main。
```
